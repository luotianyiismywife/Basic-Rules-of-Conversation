# 测试脚本使用说明

## 目录结构

```
双人/
├── 测试_example/            # 📖 测试脚本 + 示例文件
│   ├── api_client_example.py         # API 调用封装
│   ├── run_tests_example.py          # 测试运行器（入口）
│   ├── keys_example.py               # 配置模板
│   ├── 测试用例_example.py           # 测试用例清单
│   ├── 用例清单+预期结果_example.md   # 预期结果
│   ├── 角色卡-样例.md                # 角色设定格式参考
│   └── 测试脚本使用说明.md           # 本文档
│
├── 双人基础规则.md          # 规则文件（被测试引用）
└── ...
```

> **使用方式：** 复制 `keys_example.py` 为 `keys.py` 到 `测试/` 目录下（或直接放在本目录），填入 API 密钥后即可运行测试。运行器会优先使用 `测试/keys.py` 中的真实密钥。

> **已有密钥：** 如果 `双人/测试/keys.py` 已存在，运行器会自动使用它，无需额外配置。

## 一、如何编写测试用例

测试用例定义在 `测试用例_example.py` 中，格式如下：

```python
TEST_CASES = [
    {
        "id": "TC1",                          # 用例编号，唯一
        "name": "模式选择",                     # 用例名称
        "history": [                           # 对话历史（前置上下文）
            # ("角色", "内容")  角色: "user"/"assistant"
            ("assistant", "请选择模式：\n[随机模式] - ..."),
        ],
        "user_input": "指定模式",               # 最后一轮用户输入
        "note": "首次启动，AI应输出模式选择模板",  # 测试说明
    },
]
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `str` | 唯一标识，用于命令行过滤（`run_tests_example.py TC1 TC3`） |
| `name` | `str` | 显示名称 |
| `history` | `list[tuple]` | 前置对话，每个元素 `(role, content)`，`role` 为 `"user"` 或 `"assistant"` |
| `user_input` | `str` | 最后一轮的用户输入。如果为空，则 history 的最后一条必须是 user 消息 |
| `note` | `str` | 测试意图描述 |

### 添加新用例步骤

1. 在 `测试用例_example.py` 的 `TEST_CASES` 列表末尾追加新 dict
2. 编号接续（TC10、TC11...）
3. 在 `用例清单+预期结果_example.md` 中添加对应的预期输出模板和检查项
4. 运行验证：`python 双人/测试_example/run_tests_example.py TC10`

### 多轮对话测试要点

- **独立自包含**：每个用例有自己的完整 `history`，不依赖其他用例的结果
- **需要前置场景的用例**：在 `history` 中包含模式选择 + 场景设定 + AI 回复等步骤
- **AI 回复的模拟**：在 `history` 中使用的 `("assistant", ...)` 内容是人工构造的"预期回复"，不是真实的 API 输出

## 二、如何编写预期结果

预期结果定义在 `用例清单+预期结果_example.md` 中，格式如下：

```markdown
### TC3：指定模式 — 用户提供完整场景模板

**输入：** （测试用例的 user_input）

**预期输出：**
```
旁白{...}
状态{...}
情景设定完成！请发送任意内容开始互动，...
```

**检查项：**
- ✅ 用户提供的旁白+状态原文保留
- ✅ "三分糖去冰的奶茶""星露谷"等细节不得被删或模糊化
- ✅ 可以扩写，不能改写已明确的细节
```

### 检查项写作规范

- 每条检查项一个 `- ✅ / ❌` 前缀
- 明确的预期用 ✅，禁止的行为用 ❌
- 检查项是给人工审核用的，不需要自动化代码

## 三、如何运行测试

### 基本用法

```bash
# 安装依赖
pip install requests

# 从项目根目录运行（全部用例）
python 双人/测试_example/run_tests_example.py

# 只跑特定用例
python 双人/测试_example/run_tests_example.py TC1 TC3 TC5

# 跑单个用例
python 双人/测试_example/run_tests_example.py TC4
```

> **首次使用：** 复制 `keys_example.py` 为 `keys.py`，填入真实 API 密钥。
> 从 OpenCode（免费）或 DeepSeek 官方获取密钥，参考 `keys_example.py` 中的说明。

### 配置切换

编辑 `keys.py`（复制自 `keys_example.py`）：

```python
# ── API 源 ────────────────────────────────
# "opencode"  → OpenCode Go 免费接口
#               模型: deepseek-v4-flash-free
#               不支持思考模式
# "deepseek"  → DeepSeek 官方 API（需付费）
#               模型: deepseek-v4-flash
#               支持思考模式
API_SOURCE = "opencode"

# ── 思考配置（仅 DeepSeek 官方支持）────────
THINKING_ENABLED = "disabled"   # "enabled" / "disabled"
REASONING_EFFORT = "high"       # "high" / "max"
```

### API 兼容格式

| 参数 | OpenCode Go | DeepSeek 官方 |
|------|-------------|---------------|
| 费用 | 免费 | 按量付费 |
| 模型 | `deepseek-v4-flash-free` | `deepseek-v4-flash` |
| 接口地址 | `https://opencode.ai/zen/v1` | `https://api.deepseek.com/v1` |
| 密钥前缀 | `sk-...` | `sk-...` |
| 思考模式 | ❌ 不支持 | ✅ 支持 `reasoning_effort` |
| temperature | ✅ | ✅ |

### 输出位置

测试记录自动保存到 `测试记录/{配置名}/`：

| 配置 | 目录 |
|------|------|
| `THINKING_ENABLED="disabled"` | `测试记录/关闭思考/` |
| `THINKING_ENABLED="enabled"` + `REASONING_EFFORT="high"` | `测试记录/思考-high/` |
| `THINKING_ENABLED="enabled"` + `REASONING_EFFORT="max"` | `测试记录/思考-max/` |

### 批量对比运行

可通过 shell 循环或手动切换 `keys.py` 中的 `THINKING_ENABLED` 配置来对比不同思考模式下的输出差异。

```bash
# 关闭思考跑一轮
python 双人/测试_example/run_tests_example.py
# 改 keys.py 中 THINKING_ENABLED = "enabled" 后再跑一轮
python 双人/测试_example/run_tests_example.py
```

## 五、如何对比分析结果

1. 运行 `run_tests_example.py` 生成测试记录
2. 打开 `测试记录/{配置名}/` 下的文件
3. 对照 `用例清单+预期结果_example.md` 逐项检查 AI 输出
4. 关注的关键点：
   - 标签格式（旁白{}/（动作）/"台词"/记忆{}/状态{} 等）
   - 等待期 vs 阶段2 的行为差异
   - 用户提供的细节是否被保留
   - 提示语是否正确

## 六、常见问题

| 问题 | 原因 | 对策 |
|------|------|------|
| TC3 细节被改 | 模型擅自修改用户提供的细节 | 加强"扩写≠改写"规则 |
| TC4 等待期出现旁白 | 角色设定权重高于规则 | 添加铁律 + 自检清单 |
| 耗时过长 | API 限速或大模型思考 | 使用关闭思考模式 |
| 输出为空 | 网络超时 | 检查 keys.py 配置 |
